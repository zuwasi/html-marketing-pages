How It Works

Step 1 — Toolchain Detection

Auto-detects installed GNU binutils across all 7 architectures

Step 2 — Symbol Extraction

Uses nm, ar, objdump to extract global symbols from .a / .lib / .o files

Step 3 — Library Fingerprinting

Matches symbols against known patterns (Mbed TLS, OpenSSL, FreeRTOS, lwIP…) — requires ≥3 matches

Standard Binaries (ELF/PE) vs. Opaque Firmware Blobs

✅ ELF / PE / .a / .lib — SBOMator Analyzes Directly

  • • Standard headers with accessible symbol tables
  • • GNU binutils can parse and extract symbols natively
  • • Library fingerprinting identifies components (OpenSSL, Mbed TLS, FreeRTOS…)
  • • Works on .a static libraries, .o object files, ELF executables, PE/COFF binaries
  • • SBOMator handles this with full accuracy

⚠️ Opaque Firmware Blobs — Requires Unpacking First

  • • Proprietary headers (e.g., D-Link 0x81012468)
  • • Compressed payloads (gzip, LZMA) at arbitrary offsets
  • • Nested filesystems (SquashFS, CramFS) inside the blob
  • • No standard symbol tables — nm cannot parse directly
  • • Needs firmware unpacker (e.g., binwalk) to extract ELFs first
  • • Finite State specializes in this — SBOMator focuses on standard binaries

Key insight: For standard binaries (ELF/PE/static libraries), SBOMator provides precise symbol-level detection across 7 CPU architectures. For opaque vendor firmware blobs (like D-Link .had files), a firmware unpacking layer is needed first — this is Finite State's specialty.

Decision Guide — When to Choose What

Choose SBOMator When

  • ✅ You need FDA / IEC 62304 VEX with patient harm assessment
  • ✅ Your stack includes embedded / RTOS (Zephyr, nRF, FreeRTOS, Yocto)
  • ✅ You require offline / air-gapped operation
  • ✅ You want source-level SBOM precision (not binary heuristics)
  • ✅ You use Azure DevOps + Teams and want native ADO work items
  • ✅ You prefer self-hosted with no recurring SaaS costs
  • ✅ You need Conda support
  • ✅ You value supply chain threat detection (IOC database)

Choose Finite State When

  • ✅ You primarily analyze opaque firmware/binaries from third-party vendors
  • ✅ You need fleet-level risk scoring across many device variants
  • ✅ You require a mature cloud dashboard with no self-hosting
  • ✅ You use Jira as your primary issue tracker
  • ✅ You need reachability analysis to reduce CVE noise
  • ✅ Your compliance team prefers vendor-managed infrastructure
  • ✅ You need GraphQL API for complex custom integrations
SBOMator vs Finite State

Recommendation

For Azure DevOps + FDA/Embedded environments:

SBOMator delivers deeper SBOM generation, superior VEX compliance, broader ecosystem coverage, and native ADO + Teams integration — at a lower total cost of ownership with no cloud dependency.

Finite State remains the better choice only when opaque binary/firmware reverse-engineering is the primary use case and source code access is not available.
