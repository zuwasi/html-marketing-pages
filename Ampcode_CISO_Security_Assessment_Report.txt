  1. EXECUTIVE SUMMARY
================================================================================

This report provides a comprehensive cybersecurity assessment of Ampcode
(developed by Sourcegraph, Inc.) and the Sourcegraph Code Search platform to
determine their suitability for enterprise adoption within our organization.

Ampcode is an AI-powered coding agent that assists software engineers with
code generation, review, debugging, and codebase exploration. Sourcegraph is
a code intelligence platform used for cross-repository code search.

KEY FINDINGS:

  * Ampcode/Sourcegraph holds SOC 2 Type II certification and ISO/IEC
    27001:2022 certification, both independently audited.

  * The Enterprise plan provides zero data retention on ALL LLM providers —
    no customer code is retained beyond the time needed for inference.

  * Model training on customer data is permanently disabled on Enterprise
    plans and cannot be enabled.

  * All infrastructure is hosted on Google Cloud Platform in the United
    States. No infrastructure or service providers are based in China.

  * Enterprise SSO (Okta, SAML, OIDC) and SCIM Directory Sync are
    supported for centralized identity management.

  * Automatic secret redaction is built in at the system level, though it
    is best-effort and has documented limitations.

  * All data is encrypted with AES-256 at rest and TLS 1.2+ in transit.

  * Annual third-party penetration testing covers the full stack (clients,
    backend, infrastructure).

OVERALL ASSESSMENT: The security posture of Ampcode and Sourcegraph meets
enterprise requirements. Approval is recommended, conditional on Enterprise
plan activation, SSO enforcement, and developer security training.

    Ref: https://ampcode.com/security
    Ref: https://sourcegraph.com/security
    Ref: https://security.sourcegraph.com/

  4. SYSTEM ARCHITECTURE
================================================================================

Ampcode operates as a two-component architecture:

  COMPONENT 1: AMP CLIENT (Local)
  --------------------------------
  The Amp Client is the end-user application — primarily the Amp CLI
  (available via npm: @sourcegraph/amp) or integrated into VS Code.

  Responsibilities:
    - Local code and context management
    - Local settings and thread history
    - Communicates with Amp Server (ampcode.com)
    - Executes tools as requested by the server (e.g., file reads, bash)

  Security characteristics:
    - Runs entirely on the developer's local machine
    - Never clones, indexes, or stores the entire codebase
    - Best-effort avoidance of reading .env files and credential files
    - Automatic secret redaction before any data leaves the client
    - Credentials stored locally at:
        Linux/macOS: ~/.local/share/amp/secrets.json
        Windows:     %USERPROFILE%\.local\share\amp\secrets.json

  Network requirements (allowlist for corporate proxies):
    - ampcode.com
    - auth.ampcode.com
    - authapi.ampcode.com


  COMPONENT 2: AMP SERVER (Cloud)
  --------------------------------
  A multi-tenant cloud service hosted at ampcode.com on Google Cloud
  Platform.

  Responsibilities:
    - Authentication and user account management
    - Workspace management
    - Thread syncing and storage
    - Usage tracking
    - LLM inference proxy (between client and LLM providers)

  Security characteristics:
    - Hosted on GCP with PostgreSQL database (encrypted)
    - Does NOT store, clone, or index the entire codebase
    - Stores only "partial code data" — snippets of or entire code files
      that are selected as context for individual LLM requests
    - Thread data deleted within 30 days of deletion request


  DATA FLOW:

  5. DATA SECURITY & RETENTION
================================================================================

DATA SCOPE:

  The Amp Client and Amp Server do NOT see, store, clone, or index the
  entire codebase. Only "partial code data" is processed — defined as
  snippets of or entire code files that are selected as context for
  individual LLM requests. This is not an ongoing or bulk transfer.

LLM DATA RETENTION (ENTERPRISE):

  The Amp Enterprise plan (paid or trial) provides zero data retention
  for text inputs on ALL LLM providers. This means:

    - No text (including code) sent to LLMs is retained beyond the time
      required to generate an output.
    - Images are subject to extremely limited retention periods by LLM
      providers solely for compliance with law and acceptable usage
      policies (AUP).

THREAD DATA:

  Thread data includes user messages, LLM responses, code snippets used
  as context, tool call results, and attachments.

  Deletion policy:
    - When a thread is explicitly deleted, all data is removed within
      30 days of thread deletion.

  User account deletion:
    - Enterprise workspace threads: Owned by the enterprise, not the
      individual user. Threads remain when a user leaves. Data deleted
      only when the enterprise contract ends.
    - Non-enterprise workspace threads: Deleted along with user account
      per the Privacy Policy.
    - Threads outside workspaces: Deleted when user deletes account.

  Thread privacy:
    - Users can mark threads as "private" to prevent workspace sharing.
    - When joining a workspace, existing personal threads become visible
      to workspace members unless marked private or created under a
      separate account.

DATA RESIDENCY:

  All data is stored and processed in the United States.
  No infrastructure or service providers are based in China.
  For EEA/Swiss/UK data transfers, Standard Contractual Clauses are used.

PERSONAL DATA:

  Sourcegraph does not sell, share, or rent personal information for
  cross-contextual behavioral advertising (confirmed CCPA/GDPR compliant).
  Sourcegraph does not intentionally collect Sensitive Personal Information.

    Ref: https://ampcode.com/security (Data Security & Retention section)
    Ref: https://sourcegraph.com/terms/privacy

  10. AUTOMATIC SECRET REDACTION
================================================================================

Ampcode automatically detects and redacts secrets before they can enter
threads or be transmitted to any external service. This protection
operates at the lowest level of the system.

BEHAVIOR:

  When a secret is detected, it is replaced with a redaction marker:

    [REDACTED:amp]

  Detected secrets are:
    - Never visible to the LLM
    - Never stored in local cache
    - Never transmitted to LLM providers
    - Never saved on ampcode.com

SUPPORTED SECRET TYPES:

  Category              Examples
  --------------------  -------------------------------------------------
  Cloud Providers       AWS, Google Cloud, Azure credentials
  Development Platforms GitHub, GitLab, Sourcegraph, Amp access tokens
  LLM Providers         OpenAI, Anthropic, HuggingFace API keys
  Common Services       Stripe, Slack, npm tokens
  Generic Patterns      API keys, webhook secrets, password fields

  Supported secret types are regularly updated. Contact
  support@ampcode.com to request new secret types.

KNOWN LIMITATIONS:

  Secret redaction is BEST-EFFORT and may not catch:

    - Secrets in non-standard formats or custom internal systems
    - Secrets that do not follow recognizable patterns
    - Encoded or obfuscated secrets

  IMPORTANT: This is a significant consideration for organizations using
  custom or proprietary secret formats. Additional controls (developer
  training, pre-commit hooks, vault-based secret management) should be
  implemented as defense-in-depth measures.

IF A SECRET IS NOT REDACTED:

  1. Edit the message that precedes the secret being read and resend it.
     This will overwrite the thread contents on ampcode.com.
  2. Mark the thread as private (prevents workspace sharing but does not
     remove the secret from ampcode.com).
  3. Rotate the exposed secret immediately as a security precaution.

ADDITIONAL CLIENT PROTECTIONS:

  - The Amp Client makes a best-effort to avoid reading .env files and
    other credentials files.
  - Credentials are stored locally in a secure path.

    Ref: https://ampcode.com/security (Secret Redaction section)

  12. MODEL TRAINING & AI DATA USAGE POLICY
================================================================================

  - Amp does NOT train models on your data, unless you have explicitly
    opted in to training.

  - On an Enterprise workspace, turning training on requires approval
    of the Workspace Admin.

  - On an Enterprise workspace plan (paid or trial), training can
    NEVER be enabled. This is a permanent restriction.

  - Ad-supported free usage (Daily Grant) does not require opting into
    training. Enterprise workspaces can disable ad-supported free usage
    for all members.

  - When using Amp Free Mode, Sourcegraph may perform periodic automated
    keyword searches of connected files/codebase for context-specific
    advertisement placement. No personal data is collected during keyword
    searches. Results are NOT shared with advertising partners.

  RECOMMENDATION: Ensure all organization members are on the Enterprise
  plan to guarantee zero training and zero ad-supported access.

    Ref: https://ampcode.com/security (Model Training section)

================================================================================
  15. YOUR ORGANIZATION'S RESPONSIBILITIES — DETAILED GUIDANCE
================================================================================

The following section provides detailed guidance for developers and IT
teams on their security responsibilities when using Ampcode.

  A. DEVELOPER CONDUCT
  ---------------------

  AI coding agents like Ampcode can execute bash commands and read files
  on the developer's local machine. Developers MUST:

    1. NEVER paste production secrets, credentials, API keys, or PII
       directly into prompts or chat messages.

    2. REVIEW all agent-suggested commands before execution — especially:
       - rm, del (file deletion)
       - curl, wget (network requests)
       - git push, git commit (repository changes)
       - Any command involving network operations or system changes

    3. USE .env files and vault-based secret management tools. Ampcode
       makes a best-effort to avoid reading these files.

    4. ROTATE any secret that may have been inadvertently exposed in a
       thread — even if redaction was applied, treat it as potentially
       compromised.

    5. REPORT any security concerns to security@ampcode.com immediately.

    6. MARK threads as private when working on sensitive code or
       discussing security-critical systems.


  B. IT / SECURITY TEAM RESPONSIBILITIES
  ----------------------------------------

    1. Configure SSO as the EXCLUSIVE authentication method — disable
       password-based authentication for the workspace.

    2. Enable SCIM Directory Sync to automate user provisioning and
       de-provisioning aligned with your identity provider.

    3. Establish a secret rotation playbook specifically for scenarios
       where secrets may have been exposed through LLM context.

    4. Allowlist the following domains on corporate proxy/firewall:
         - ampcode.com
         - auth.ampcode.com
         - authapi.ampcode.com

    5. Periodically request and review application audit logs from
       Ampcode (available for Enterprise workspaces upon request).

    6. Verify that the Enterprise plan is active and training is
       permanently disabled.

    7. Disable Daily Grant (ad-supported free usage) for all workspace
       members if ad-related keyword scanning is not acceptable.

    8. Review the subprocessor list periodically:
       https://sourcegraph.com/terms/subprocessors

    9. Subscribe to subprocessor change notifications.


  C. PROMPT INJECTION AWARENESS
  ------------------------------

  Prompt injection is an inherent risk with all LLM-based tools.
  Ampcode's bug bounty program accepts prompt injection reports but
  notes they are not typically eligible for bounty rewards due to the
  inherent nature of LLMs. Developers should:

    - Be cautious when the agent reads files from untrusted sources
    - Review any unusual or unexpected commands suggested by the agent
    - Never blindly execute multi-step commands without review
    - Report suspicious agent behavior to security@ampcode.com
