Data Security

Data Retention & Training Policies

How each vendor handles customer code data, retention periods, and whether data is used for model training.

🟢

Ampcode — Zero Retention

  • Enterprise: zero retention on ALL LLM providers
  • Training permanently disabled, cannot be re-enabled
  • Thread data deleted within 30 days of request
  • Images: limited retention for compliance only

Best-in-Class
🔴

Cursor — Mixed Modes

  • Privacy Mode: zero retention (must be enforced)
  • Share Data mode: data MAY be used for training
  • Already-trained models NOT retrained after deletion
  • Training data persists permanently in models

Caution Required
🔵

Claude Code — API Separation

  • Enterprise / API: no training on customer data
  • Consumer: may train unless user opts out
  • CLI-local: code never stored on intermediate server
  • Clear enterprise vs. consumer boundary

Strong
🟡

Codex — ZDR on Request

  • Default: 30-day abuse monitoring retention
  • Zero Data Retention requires approval process
  • Not all API endpoints eligible for ZDR
  • Additional requirements must be met for ZDR

Adequate
🚨
Cursor Training Risk: If Privacy Mode is not enforced from day one, code may be used for model training. Once data has been incorporated into model weights, deletion requests cannot remove it. This creates a permanent, irrecoverable data exposure.

Secret Protection & Sandboxing

How Each Tool Protects Secrets & Isolates Execution

Secret Protection

BEST
Ampcode — Automatic Redaction

System-level engine detects AWS, GCP, Azure, GitHub, GitLab, OpenAI, Anthropic, Stripe, Slack, npm tokens + generic patterns. Replaces with [REDACTED:amp]. No developer config needed.

WEAK
Cursor — No Redaction Engine

Only .cursorignore files (developer-configured). No automatic detection. Extension sigs DISABLED by default. Workspace Trust DISABLED by default. Entire burden on developers.

STRONG
Claude Code — Defense in Depth

Encrypted cred storage, command blocklist (curl/wget blocked), command injection detection, sandboxed bash with filesystem/network isolation.

STRONG
Codex — Platform Sandbox

Platform-native enforcement in sandboxed cloud environments. Network isolation configurable. Approval policies for sensitive operations.

Sandboxing & Execution Isolation

Vendor Model Score
Claude Code Permission-based, sandboxed bash, fs/net isolation, write-only workdir 5/5

⚠️ Shared Responsibility Model

What Vendors Secure vs. What You Must Secure

Regardless of which tool you deploy, your organization retains critical security responsibilities.

🟢 Vendor Responsibility (All Tools)

  • Application security (secure SDLC, code reviews, pentesting)
  • Infrastructure security (cloud hardening, firewall, WAF)
  • Encryption at rest (AES-256) and in transit (TLS 1.2+)
  • Compliance certification maintenance
  • Security patch management and vulnerability remediation
  • Incident response and security monitoring
  • LLM provider agreement enforcement

🟡 Your Organization's Responsibility

  • Tool selection — choose tools matching your risk tolerance
  • Access management — SSO config, user lifecycle, offboarding
  • Privacy mode enforcement — ensure training is disabled
  • Secret hygiene — vault usage, rotation, .cursorignore / .env
  • Developer training — AI agent risks, prompt injection awareness
  • Code review — review all AI-generated code before merge
  • CVE monitoring — track vendor vulnerability disclosures
  • Version management — pin and update tool versions
  • Incident response — playbooks for secret exposure scenarios
  • Compliance — understand regulatory impact of AI tool usage
🚨
Critical for ALL Tools: AI coding agents can execute commands and read files on developer machines. Developers must: Never paste production secrets into prompts · Review agent-suggested commands before execution · Rotate any secret exposed in a session · Report security concerns immediately
