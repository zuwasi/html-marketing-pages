Should We Fork Token Optimizer for Amp?

Verdict: No Fork. Build an Amp-Native Skill Instead.

❌ Why a Fork Is Wrong

  • 75% of features solve problems Amp doesn't have
  • Deeply coupled to Claude Code's hook system (PreToolUse, PreCompact, etc.)
  • Python stdlib design ≠ Amp's Node/Bun ecosystem
  • Maintaining a fork of 271-commit project = ongoing burden
  • OpenClaw port proved: each platform needs a native rewrite

✅ What WOULD Be Valuable for Amp

  • Session analytics dashboard — per-turn token/cost visibility
  • Quality scoring — adapted 5-signal score
  • Loop detection — catch retry spirals
  • Delta mode concept — worth exploring
  • Build as Amp skill with bundled MCP server

💡 Recommendation

Don't fork. Build amp-context-health — a lightweight Amp skill that cherry-picks the 3–4 features that actually apply. Ship as a skill + MCP server. ~2-3 days vs. 4-6 weeks maintaining a fork of irrelevant code.

Proposed: amp-context-health Skill

Cherry-pick what matters, ignore what doesn't

📊 Session Dashboard

Per-turn token counts, cost tracking, cache hit ratios. Auto-opens after session via local HTML.

Effort: ~6 hours

🔄 Loop Detector

Compare last N messages for similarity. Inject short note at confidence ≥ 0.7. Saves 10-50K tokens per caught loop.

Effort: ~3 hours

📈 Thread Health Score

5-signal metric: Context fill (25%), Stale reads (25%), Bloated results (20%), Thread age (15%), Decision density (15%)

Effort: ~4 hours

🤖 Handoff Advisor

Unique to Amp: detect when it's time to handoff. "Context at 72% with quality B — consider handoff."

Effort: ~2 hours

Bottom Line

Amp wins for 90% of real-world developer workflows — prevents the problems rather than managing them. Token Optimizer wins for the 10% who need deep session visibility or prefer long exploratory sessions.

The fact that TO exists with 300+ ⭐ proves there's real pain. The right response is to solve it architecturally — which is exactly what Amp did.

Summary & Recommendations

✅ Key Takeaways

  • Context decay is a universal LLM problem — every tool faces it
  • Two valid approaches: prevent by design (Amp) vs. manage with tooling (TO)
  • Amp's handoff model maintains peak quality throughout any length session
  • Token Optimizer is excellent engineering for a problem Amp doesn't have
  • 75% of TO's features are irrelevant in Amp's architecture

🚀 Recommended Actions

  • Don't fork Token Optimizer — incompatible architecture
  • Do build amp-context-health as a native Amp skill
  • Cherry-pick: dashboard, loop detection, quality scoring, handoff advisor
  • Effort: 2-3 days vs. 4-6 weeks for a fork
  • Ship as skill + MCP server for zero-overhead integration
